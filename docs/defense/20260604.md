## 3. Como explicar el diagrama entidad-relacion

El ERD esta en:

```text
docs/database/erd.md
```

La explicacion simple:

> El diagrama muestra como se organizan los datos del negocio. Cada tabla
> representa una entidad importante y las lineas muestran como se relacionan.
> Las claves primarias identifican cada registro y las claves foraneas conectan
> una tabla con otra.

### Relaciones que debo dominar

#### `PERSON` y `CUSTOMER`

`PERSON` guarda datos personales:

- nombre
- apellido
- carnet o documento
- telefono
- correo
- direccion

`CUSTOMER` guarda el rol comercial de esa persona como cliente.

Respuesta lista:

> Separe `PERSON` y `CUSTOMER` porque una persona es la identidad civil, y
> cliente es el rol dentro del negocio. Asi no duplico datos personales si en el
> futuro esa persona cumple otro rol.

#### `PART_CATEGORY`, `PART` e `INVENTORY_STOCK`

`PART_CATEGORY` clasifica repuestos.

`PART` guarda el repuesto:

- codigo interno
- codigo OEM
- nombre
- marca
- precio de venta
- costo de compra
- estado

`INVENTORY_STOCK` guarda el stock actual del repuesto.

Respuesta lista:

> No puse el stock directamente como unico dato del repuesto porque el repuesto
> y su existencia fisica son conceptos diferentes. `PART` describe el producto;
> `INVENTORY_STOCK` controla cuantas unidades hay y donde estan.

#### `SALES_ORDER` y `SALES_ORDER_ITEM`

`SALES_ORDER` es la cabecera de la venta.

`SALES_ORDER_ITEM` es el detalle de los productos vendidos.

Respuesta lista:

> Una venta puede tener varios repuestos, por eso uso una tabla de cabecera y
> una tabla de detalle. Si pusiera todos los productos en una sola columna, la
> base dejaria de ser relacional y seria dificil consultar o validar.

#### `PAYMENT`

`PAYMENT` se relaciona con `SALES_ORDER`.

Respuesta lista:

> Separe el pago de la venta porque una venta y un pago no son exactamente lo
> mismo. La venta guarda que se vendio y cuanto cuesta. El pago guarda como y
> cuando se cancelo.

#### `PART_COMPATIBILITY`

En el ERD completo, `PART_COMPATIBILITY` conecta repuestos con modelos de
vehiculos.

Respuesta lista:

> Esta tabla resuelve una relacion muchos-a-muchos: un repuesto puede servir
> para varios vehiculos y un vehiculo puede aceptar varios repuestos.

## 4. Que son las rutas `@app.route`

Archivo:

```text
src/zarvent_repuestos/web/app.py
```

Explicacion humana:

> `@app.route` le dice a Flask que funcion debe ejecutarse cuando el usuario
> entra a una direccion de la web. Es como un mapa: si el usuario entra a
> `/inventory`, Flask llama a la funcion `inventory()`.

No debo decir que `@app.route` es "la conexion entre templates". Mejor decirlo
asi:

> La ruta conecta una URL con una funcion de Python. Esa funcion puede leer
> formularios, llamar a la base de datos y devolver un template HTML.

### Rutas actuales del proyecto

| Ruta | Funcion | Que hace |
| --- | --- | --- |
| `/` | `home()` | Muestra login y procesa usuario/contrasena. |
| `/logout` | `logout()` | Limpia la sesion y vuelve al login. |
| `/dashboard` | `dashboard()` | Muestra metricas de ventas, ordenes y stock bajo. |
| `/inventory` | `inventory()` | Lista repuestos y permite registrar un nuevo repuesto. |
| `/sales` | `sales()` | Lista ventas y permite crear una nueva venta. |
| `/sales/receipt/<int:sales_order_id>` | `receipt()` | Muestra el recibo de una venta especifica. |

### Ejemplo para explicar `/inventory`

> Cuando entro por GET a `/inventory`, la pagina lista los repuestos usando
> `part_crud.listar_productos()`. Cuando envio el formulario por POST, Flask lee
> los datos del formulario, crea un objeto `Part` y llama a
> `part_crud.crear_producto(...)`. Esa funcion hace los INSERT en MySQL.

### Ejemplo para explicar `/sales`

> Cuando entro por GET a `/sales`, la pagina lista ventas, repuestos y clientes.
> Cuando envio una venta por POST, Flask lee el cliente, el metodo de pago y los
> items vendidos. Luego llama a `sales_crud.crear_orden_venta(...)`, que guarda
> la venta, guarda sus items, registra el pago y descuenta el stock.

## 5. Como explicar los queries SQL dentro de Python

Los queries no estan todos escritos directamente en `app.py`. La app esta
separada por capas:

```text
app.py
  -> llama funciones CRUD
crud/
  -> tiene los INSERT, SELECT, UPDATE y DELETE
connection.py
  -> abre la conexion a MySQL
MySQL
  -> guarda los datos
```

Respuesta lista:

> En este proyecto `app.py` no mezcla toda la logica SQL. `app.py` recibe la
> peticion web y llama a funciones CRUD. Las consultas SQL estan principalmente
> en `part_crud.py`, `customer_crud.py`, `sales_crud.py` y `user_service.py`.

### Tipos de query que debo reconocer

| Tipo | Ejemplo del proyecto | Para que sirve |
| --- | --- | --- |
| `SELECT` | Buscar usuario, listar repuestos, listar ventas. | Leer datos. |
| `INSERT` | Crear usuario, producto, cliente, venta, item y pago. | Agregar datos. |
| `UPDATE` | Descontar stock despues de una venta. | Modificar datos existentes. |
| `DELETE` | Borrar usuario o producto desde funciones CRUD. | Eliminar datos. |

### Por que se usa `%s` en los queries

En el codigo se usan queries parametrizados:

```python
cursor.execute(query, (username,))
```

Respuesta lista:

> Uso parametros `%s` para no pegar texto del usuario directamente en el SQL.
> Eso es mas seguro y evita errores con comillas o datos raros.

### Que es `commit()`

Respuesta lista:

> `commit()` confirma los cambios en la base de datos. Si hago un INSERT o
> UPDATE y no confirmo, el cambio puede no quedar guardado.

### Que es `rollback()`

Respuesta lista:

> `rollback()` revierte los cambios cuando ocurre un error. Sirve para que la
> base no quede a medias.

### Que es una transaccion

Ejemplo fuerte del proyecto: crear una venta.

En una venta se hacen varias operaciones:

1. Verificar stock.
2. Insertar `sales_order`.
3. Insertar `sales_order_item`.
4. Descontar `inventory_stock`.
5. Insertar `payment`.
6. Confirmar con `commit()`.

Respuesta lista:

> Una transaccion agrupa varias operaciones para que se ejecuten completas o no
> se ejecuten. En una venta, no quiero que se guarde la orden pero no se
> descuente stock. Por eso si algo falla, se hace `rollback()`.

### Que significa `SELECT ... FOR UPDATE`

En `sales_crud.py`, antes de vender se consulta el stock con bloqueo:

```sql
SELECT quantity_on_hand FROM inventory_stock WHERE part_id = %s FOR UPDATE
```

Respuesta lista:

> `FOR UPDATE` bloquea temporalmente esa fila mientras se procesa la venta. Asi
> evita que dos ventas descuenten el mismo stock al mismo tiempo.

## 7. Demo 1: registrar un repuesto y verificar en MySQL

### En la web

1. Entrar a `Inventory`.
2. Registrar un repuesto nuevo.
3. Usar un `internal_code` facil de encontrar, por ejemplo:

```text
DEF-20260604-01
```

4. Guardar.
5. Verificar que aparece en la lista.

### En MySQL

Entrar a MySQL y ejecutar:

```sql
USE sis122_zarvent_repuestos;

SELECT part_id, internal_code, name, brand, sale_price, purchase_cost, status
FROM part
WHERE internal_code = 'DEF-20260604-01';

SELECT s.inventory_stock_id, s.part_id, s.quantity_on_hand, s.location_name
FROM inventory_stock s
JOIN part p ON s.part_id = p.part_id
WHERE p.internal_code = 'DEF-20260604-01';
```

Que debo explicar:

> Al registrar un repuesto, no solo se inserta en `part`. Tambien se crea su
> stock inicial en `inventory_stock`. Eso demuestra la relacion entre producto e
> inventario.

### En la web

### En MySQL

Ver ultimas ventas:

```sql
USE sis122_zarvent_repuestos;

SELECT sales_order_id, customer_id, order_date, status, subtotal, total_amount
FROM sales_order
ORDER BY sales_order_id DESC
LIMIT 5;
```

Ver items de la ultima venta:

```sql

       i.unit_price, i.discount_amount
FROM sales_order_item i
ORDER BY i.sales_order_item_id DESC
LIMIT 10;
```

Ver pagos:

```sql
SELECT payment_id, sales_order_id, payment_date, method, amount, status
FROM payment
ORDER BY payment_id DESC
LIMIT 5;
```

Ver stock:

```sql
SELECT p.internal_code, p.name, s.quantity_on_hand, s.reorder_level
FROM inventory_stock s
JOIN part p ON s.part_id = p.part_id
ORDER BY s.inventory_stock_id DESC
LIMIT 10;
```

Que debo explicar:

> Una venta afecta varias tablas. Se crea la cabecera en `sales_order`, el
> detalle en `sales_order_item`, el pago en `payment` y se descuenta el stock en
> `inventory_stock`.

## 10. Lo que debo decir si pregunta por CRUD

CRUD significa:

| Letra | Significado | En el proyecto |
| --- | --- | --- |
| C | Create | Crear repuestos, clientes, ventas, pagos. |
| R | Read | Listar dashboard, inventario, ventas y clientes. |
| U | Update | Descontar stock al crear una venta; funciones para actualizar producto y usuario. |
| D | Delete | Funciones para borrar producto y usuario. |

Importante para no exagerar:

> La web actual muestra principalmente Create y Read, y tambien hace Update de
> stock cuando se registra una venta. En el codigo existen funciones de Update y
> Delete para producto y usuario, pero no todas tienen pantalla web dedicada.

Si el profesor pide especificamente "dar de baja usuarios, cambiar roles o
asignaciones", debo responder con honestidad:

> En mi prototipo actual el login usa usuarios, pero no implemente una pantalla
> administrativa de roles. Para Base de Datos I priorice el flujo relacional de
> clientes, repuestos, inventario, ventas y pagos. Como mejora futura se puede
> agregar una tabla de roles y rutas web para administrarlos.

Eso es mejor que inventar una funcion que no existe.

## 12. Preguntas probables y respuestas simples

### Que es Flask?

> Flask es un framework de Python para crear aplicaciones web. En mi proyecto
> recibe las peticiones del navegador, procesa formularios y devuelve paginas
> HTML.

### Que es MySQL Connector?

> Es la libreria que permite que Python se conecte a MySQL, ejecute consultas y
> reciba resultados.

### Para que sirve `bcrypt`?

> Sirve para guardar contrasenas de forma segura. No se guarda la contrasena en
> texto plano; se guarda un hash.

### Por que usas claves foraneas?

> Porque aseguran que una relacion tenga sentido. Por ejemplo, una venta debe
> pertenecer a un cliente real y un item de venta debe apuntar a un repuesto
> real.

### Por que algunas claves tienen `ON DELETE RESTRICT`?

> Porque no quiero borrar un dato si todavia esta siendo usado. Por ejemplo, si
> un repuesto ya fue vendido, no deberia eliminarse y romper el historial.

### Por que algunas claves tienen `ON DELETE CASCADE`?

> Porque hay datos dependientes que no tienen sentido solos. Por ejemplo, si se
> borra una orden, sus items dependientes tambien deberian borrarse.

### Por que `SALES_ORDER_ITEM` guarda `unit_price`?

> Porque el precio del repuesto puede cambiar despues. Si no guardo el precio
> historico, una venta antigua podria cambiar de valor cuando actualizo el
> precio del producto.

### Que pasa si no hay stock suficiente?

> La funcion de venta valida el stock antes de insertar la venta. Si no hay
> suficiente, lanza error y no confirma la transaccion.

### Que pasa si falla una venta a la mitad?

> Se hace `rollback()`. Eso evita que quede una venta incompleta o un stock mal
> descontado.

### Donde se crean las tablas?

> Para la demo, las tablas se crean en
> `src/zarvent_repuestos/database/init_db.py`. El archivo
> `database/schema.sql` es el borrador manual academico basado en el ERD.

### Donde estan las consultas?

> Las consultas estan principalmente en `src/zarvent_repuestos/crud/` y en
> `src/zarvent_repuestos/access/user_service.py`. `app.py` coordina la web y
> llama a esas funciones.
